Component guide

How the Scintilla app works

A tour of every screen and component — what each one is for, how the pieces fit together, and how the interactive parts behave. Read it top to bottom, or jump to any component on the left.

Overview

Scintilla turns The Experimentation Field Book's five-step process and fourteen templates into a working tool for practitioners and design-thinking students. It has two halves that share the same data, plus a dedicated systems-thinking workspace:

The dashboard

Portfolio at a glance — every concept, its stage in the process, active tests, and evidence.

The guided workflow

Open a concept and work through Steps 1–5, filling in each step's templates as you go.

Note

The four case studies from the book seed the demo data — Easykicks (Nike), Welcome Host (South Western Railway), Career Navigator (PMI) and ER Fast-Track (Whiteriver Hospital). Concepts are fully yours to manage: use New concept in the top bar or Concepts screen to create one, click any concept to open it, and hover a row in the Concepts list to reveal a delete button.

Anatomy of the app

Every screen shares the same chrome: a fixed sidebar on the left, a top bar, and a scrolling content area. Only the content area changes as you navigate.

SIDEBAR
TOP BAR
CONTENT AREA
  • Sidebar — brand lockup plus two nav groups: Workspace (Dashboard, Concepts, Guided Workflow, Tests, Progress) and Systems (Systems maps, badged "NEW"). The active item is a solid blue pill. Below 900 px it collapses to an icon-only rail.
  • Top bar — a page title and subtitle that update per screen, a search field, a notification bell, and the primary New concept button.
  • Content area — the only thing that swaps. Each screen fades and rises in over ~0.3 s.

Component reference

Here's each screen in turn — what it's for, what's in it, and how it behaves.

Dashboard & its variants

The home screen. A segmented toggle switches between three layout directions so you can pick the one that fits how you work:

  • A · Mission control — four KPI cards over a two-column grid: a concept portfolio list (each row shows a five-segment step-progress bar) beside an "Up next" task list.
  • B · Pipeline — a kanban with one column per step; concept cards flow left to right as they advance.
  • C · Editorial — a dark hero spotlighting the concept in the field right now, with big headline metrics and three supporting widgets.
A · Mission control B · Pipeline C · Editorial
Tip

In production, ship Variant A as the default and treat B and C as optional home layouts — they exist so you can compare directions before committing.

Concepts & the Value/Effort Matrix

This is Step 1's home. The signature component is the Value/Effort Matrix: a 2×2 with a value y-axis and an effort x-axis, split into Quick Wins, Strategic Bets, Fill-ins and Avoid. Each concept is a draggable pill positioned by its value/effort scores; the top-left "quick win" is highlighted. Beside it, an "All concepts" list gives a linear, scannable view — shelved concepts appear dimmed.

Workspace & the Stepper

Opening a concept enters the guided workflow. A concept header sits above the Stepper — five connected nodes for the five steps. The active node is a blue ring; completed steps show a green check. Clicking a node, or the Back / Continue buttons at the foot of each step, moves you through the flow. The header's name, organization chip, and one-line description are all click-to-edit.

The step templates

Every box in every step is a real, per-concept record — click any label or value to edit it in place (Enter or click away commits, Escape cancels). A few highlights:

  • Concept Snapshot & Storyboard — editable rows and user-column labels; click a storyboard frame to toggle it blank (for co-creation) versus filled, and edit its caption underneath.
  • Surfacing Assumptions — three colour-coded columns for Desirability (blue), Feasibility (teal) and Viability (slate); add or delete assumption chips per column.
  • Assumptions → Evidence — a table pairing each assumption with observable evidence, a threshold→aspirational target, and a source; add or delete rows.
  • Test Decision Flow & Test Digest — the three funnel questions drive a live recommendation; a "Use in Test Digest" shortcut fills the recommended test type, and every Test Digest cell is editable. Confirm test logs the current recommendation and digest straight into the Tests tracker as "In field."
  • Research Guide & Test Audit Checklist — click any item to check/uncheck it.
  • Results — threshold-versus-actual bars with PASS / BELOW pills computed live from editable actual/threshold values; add or delete result rows, and edit the live-status note and the Iterate write-up.

Tests & Progress

Tests is a cross-concept tracker — a table of every test with its type, Say/Do character, status and result, populated by whatever you've confirmed from Step 3 across all your concepts. Progress renders the active concept's Progress Tracker as a vertical timeline of the five steps (done, current, upcoming), plus mini progress bars for up to three other concepts.

Systems mapping

A dedicated Systems maps workspace helps you understand the system an idea lives in before you test it. A tab switcher holds four tools — the first three are real, editable tools, not illustrations:

  • System Support Map — a role-centred radial map. Your role sits at the centre, ringed outward by Responsibilities → Needs → Resources → Wishes. "Add to the map" creates a real sticky note on the ring you pick; click a note to rename it, drag to reposition, hover to reveal delete, and click a resource's star to cycle helpful / neutral / unhelpful. Every note persists per concept.
  • Feedback Loops — a real causal-loop canvas built on React Flow. "+ Element" adds a node; "+ Link" walks you through picking a source, a target, and a +/− polarity. Hover an item in the Elements or Links lists on the right to reveal a delete button, or use Clear in the toolbar to wipe the canvas and start over. The canvas auto-detects every closed loop and labels it R (reinforcing, blue) or B (balancing, teal) by counting − links around the cycle — even is reinforcing, odd is balancing. A right-rail panel explains loop directionality (goal, gap, corrective action, delay) alongside the rule the app itself uses.
  • 5Rs System Diagnostic — a library of named, timestamped assessments of a local system's "as is" state, adapted from USAID's 5Rs Framework. "New diagnostic" opens a blank record; rate each of Results, Roles, Relationships, Rules and Resources 0–5, answer that element's guiding questions (drawn verbatim from the framework's Table 1 — 6/7/5/4/5 questions respectively), and note any gaps. A radar chart summarises all five ratings at a glance. "Mark as final" locks a diagnostic to view-only; "Duplicate" branches a new draft from any entry, final or not.
  • Systems Archetypes — a gallery of the classic traps (Fixes that Fail, Limits to Growth, Tragedy of the Commons, Shifting the Burden…). Each card opens a detail view with the pattern, a "watch for it when…" cue, where to intervene, and a behaviour-over-time sketch.
Tip

The loops and archetypes are linked — the Feedback Loops insight card names the detected R and B loops and jumps straight to the matching archetype so you know where to push. Support Map, Feedback Loops and the 5Rs Diagnostic can all pop out into their own larger window (top-right of each tool) and export straight to PDF.

Interactive pieces

Almost nothing in Scintilla is a static form — concepts, step templates, and every systems tool read from and write to real, per-record state. A few pieces worth calling out specifically:

Test Decision Flow

Three questions funnel down to a single recommended test type. The recommendation panel recomputes live as you answer:

if   Q1 == "yes"                         // data already exists   → Thought test
elif Q3 == say  and Q2 == component     → Cognitive walkthrough
elif Q3 == say  and Q2 == whole         → Lemonade stand
elif Q3 == do   and Q2 == component     → Smoke test
elif Q3 == do   and Q2 == whole         → Simulation / Trial

Say/Do & Fidelity sliders

Native range inputs (tinted with the brand blue) capture where a test sits on the Say↔Do and low↔high-fidelity continuums. Their values persist between sessions.

The Matrix & the Stepper

The Value/Effort matrix positions concepts by stored coordinates (drag to re-score); the Stepper drives which step body is shown. Both read from and write to the same app state described next.

Data & autosave

The prototype keeps everything in local app state — no backend required. A small set of values drives navigation and the interactive pieces, and all of it is written to localStorage on every change and restored on load, so a refresh never loses your place.

PERSISTED STATE · fb_app_state_v1
screen — dashboard | concepts | workspace | tests | progress | systems
home — a | b | c // dashboard variant
step — 1 … 5 // active workspace step
concepts — // every concept you've created, renamed or deleted
activeConceptId — // which concept the workspace/Progress screens show
workspaceData — per-concept Steps 1–5 // {step1…step5}, keyed by concept id
tests — // every confirmed test across all concepts
sysTab — support | loops | fiveRs | archetypes // systems map tool
sysArch — archetype id | null // open detail card
supportMaps — per-concept support-map notes // {id, ring, text, x, y, star?}
loopGraphs — per-concept loop graphs // {nodes[], links[]}
fiveRs — the 5Rs diagnostic library // {id, name, status, elements: {rating, answers[], gapNote}}
activeFiveRId — id | null // open diagnostic, or the library list

Concepts, per-concept workspace data, tests, and each systems map (support maps, loop graphs, and 5Rs diagnostics) are modelled as record collections, so the seeded case-study data lives side by side with real, user-created records — nothing has to be swapped out in the UI.
